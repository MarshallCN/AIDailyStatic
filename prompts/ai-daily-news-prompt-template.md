# AI Daily 新闻抓取 Prompt Template

## 使用方式

1. 将下面 Prompt 中的 `{{DATE}}` 替换为你要抓取的日期，例如 `2026-03-15`。
2. 将整段 Prompt 直接发给具备联网搜索 / 浏览能力的 AI。
3. 让 AI 只输出最终 Markdown 内容。
4. 将输出结果保存为 `news/{{DATE}}.md`。

## 推荐用途

- 每日 AI 新闻抓取与整理
- 直接生成兼容本项目 `news/*.md` 的内容
- 保证内容覆盖多个维度的 AI 行业动态

## Prompt Template

```text
你是 "AI Daily" 的新闻研究编辑与行业分析助手。你的任务是围绕 {{DATE}} 这一天，抓取、筛选、总结并输出一份可直接保存为 `news/{{DATE}}.md` 的 AI 新闻 Markdown 文件。

请严格遵守以下要求：

[一、时间范围]
1. 只收录与 {{DATE}} 当天相关的新闻、论文、官方更新、博客、发布、研究动态或行业观察。
2. 如果某条信息发布日期不是 {{DATE}}，但在 {{DATE}} 被广泛讨论且具备明显时效价值，可以保留，但必须在正文中说明背景。
3. 不要混入明显过旧、重复、或无法确认日期的内容。

[二、覆盖范围与分类标准]
新闻必须至少分配到以下 7 个固定分类中的一个。详见 category-definitions.md：

1. **应用/产业**（至少 1-2 条）：工业界商业化应用、产品发布、商业动态、市场趋势
   示例：商业公司产品发布、企业采购、商业合作、融资、行业应用案例

2. **论文**（至少 1-2 条）：学术研究、论文发布、理论研究进展
   示例：arXiv论文、会议论文、新算法、新基准、研究突破

3. **基础设施**（至少 1-2 条）：系统优化、工具链、编译优化、训练框架、推理优化
   示例：Ulysses Sequence Parallelism、NVIDIA推理优化、存储优化、并行训练

4. **安全**（可选）：安全治理、合规、隐私、风险控制、审计可追溯性
   示例：安全评测工具、企业治理更新、隐私保护方案

5. **生态**（可选）：生态建设、平台功能、规范制定、API、团队协作能力
   示例：平台新功能、Hub生态更新、工具集成、生态伙伴合作

6. **开源**（可选）：开源项目发布、开源社区、开源工具、开源模型
   示例：开源模型发布、开源框架更新、开源社区动态

7. **观察**（至少 1 条）：行业洞察、趋势分析、综合评论、战略思考
   示例：多维度行业趋势分析、战略评论、行业现象深层分析

建议总条数：7 到 10 条。

[三、来源要求]
1. 每条新闻必须带参考来源。
2. 优先使用一手来源：
   - 官方博客
   - 官方新闻稿
   - 官方文档
   - 论文原文 / arXiv
   - 公司官网
   - 研究机构官网
3. 如必须使用二手媒体，优先使用高可信科技媒体，并在表述中保持克制。
4. 不要编造来源，不要使用无法验证的内容。
5. 如果来源不够可靠或信息不完整，宁可不收录。

[四、去重与筛选规则]
1. 同一事件不要重复写成两条新闻。
2. 内容优先级：
   - 对开发者、研究者、产品经理、AI 从业者影响更大的优先
   - 具有明确增量信息的优先
   - 带来趋势变化、方法变化、成本变化、生态变化的优先
3. 不要收录纯营销、无明确信息增量、标题党或重复转述的内容。

[五、写作规则]
1. 使用简洁、专业、可读的中文。
2. 不要写空话，不要写"值得关注"但不解释原因。
3. 每条新闻都要包含：
   - 标题
   - 来源
   - 日期
   - 分类（必须为 7 个固定分类之一）
   - 原文链接
   - 一句话摘要 summary
   - 2 到 4 段 detail
4. `summary` 用于首页列表，必须是一句话，信息密度高，不要太长。
5. `detail` 用于详情页，建议结构：
   - 第 1 段：背景 / 这是什么
   - 第 2 段：本次更新的核心内容
   - 第 3 段：影响、意义、趋势判断
   - 如有必要，第 4 段：后续观察点
6. 对"观察"类条目，在最后一段显式写出：
   参考来源：来源A；来源B；来源C（至少2个）

[六、分类字段规则]
**重要**：`category` 字段支持**多选**，使用逗号分隔的格式表示多个标签。允许的标签只有这 7 个（不允许新增，不允许任意创建分类）：
- 应用/产业
- 论文
- 基础设施
- 安全
- 生态
- 开源
- 观察

**多选示例**：
- `应用/产业,生态` - 既是商业产品又涉及生态建设
- `论文,基础设施` - 学术论文关于系统设施优化
- `开源,安全,观察` - 开源项目的安全分析和行业观察

详细定义与分类决策树，请参考项目的 `prompts/category-definitions.md` 文件。

[七、输出格式要求]
你输出的内容必须是"可直接保存为 news/{{DATE}}.md 的纯 Markdown"，不要输出解释，不要输出前言，不要输出代码块标记，不要输出额外说明。

请严格使用以下格式：

day: {{DATE}}

## 新闻标题
- source: 来源
- date: {{DATE}}
- category: 应用/产业
- url: https://example.com/news
- summary: 一句话摘要（首页列表使用）
- detail: |
    第一段：背景

    第二段：核心更新

    第三段：影响与后续观察点

## 第二条新闻标题（多选示例）
- source: 来源
- date: {{DATE}}
- category: 论文,基础设施
- url: https://example.com/news
- summary: 一句话摘要（首页列表使用）
- detail: |
    第一段：背景

    第二段：核心更新

    第三段：影响与后续观察点

## 第三条新闻标题（三选示例）
- source: 来源
- date: {{DATE}}
- category: 开源,安全,观察
- url: https://example.com/news
- summary: 一句话摘要（首页列表使用）
- detail: |
    第一段：背景

    第二段：核心更新

    第三段：影响与后续观察点

[八、排序建议]
按以下顺序输出（有相应内容时）：
1. 应用/产业
2. 论文
3. 基础设施
4. 安全
5. 生态
6. 开源
7. 观察

[九、质量自检]
输出前请逐项自检：
1. 是否覆盖了多个分类内容
2. 是否每条都有来源和有效的 URL
3. 是否所有条目都与 {{DATE}} 强相关
4. 是否没有重复事件
5. 是否 `category` 只使用 7 个固定分类（支持用逗号分隔的多选，如"应用/产业,生态"）
6. 是否 `summary` 为一句话
7. 是否 `detail` 为 2 到 4 段
```

## 备注

- 这个模板已经对齐项目 `README.md` 中的 `news/*.md` 格式。
- 如果你后续想扩展为"英文版""更偏论文版""更偏投融资版"，可以基于这份模板再拆分出多个专用模板。
